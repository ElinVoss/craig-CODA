A. Briefing Desk

Default landing screen. Top half is the mandatory state report. Lower half lets you append decisions and edit live vault notes.

Pros

  • Forces context first
  • Best handoff to future models
  • Still keeps editing close by

Cons

  • Less file-system-like than a pure vault browser

folder tree

B. Vault Explorer

Feels closest to Obsidian. Browse folders and `_method.md` notes first, with the report living beside the editor.

Pros

  • Strong vault feel
  • Great for browsing parent-child method chains

Cons

  • The briefing can become secondary
  • Easier to skip the state report

C. Mission Log

Report-first to the extreme. The main screen is a chronological operating record, with editing moved behind an action button.

Pros

  • Strongest audit trail
  • Best for project memory

Cons

  • Editing becomes slower
  • Less like a live vault workspace

Recommendation: A. Briefing Desk.

It satisfies your strongest requirement first: the next model cannot miss the current state. Then it still lets you edit the live vault notes that drive corpus, tokenizer, architecture, and weight behavior.
