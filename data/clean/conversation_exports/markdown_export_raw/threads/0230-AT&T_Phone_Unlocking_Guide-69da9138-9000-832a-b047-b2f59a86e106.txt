Here’s the stack, bottom to top:

---

## 1) Baseband / Modem firmware (lowest level)
This is the Qualcomm modem subsystem.

- Runs separately from Android
- Handles SIM authentication, LTE/5G registration

**Where lock shows up here:**
- A flag in modem NV (non-volatile) configuration:
  - “SIM lock enabled”
  - Allowed MCC/MNC list (carrier codes like AT&T’s network IDs)

If SIM doesn’t match allowed network:
- Modem refuses proper registration or limits service

---

## 2) SIM lock policy / NV items
Stored inside modem “NV memory” (non-volatile settings).

Typical contents:
- Allowed carrier IDs
- Lock status (locked/unlocked)
- Attempt counters (for NCK entry)

This is where the **actual “locked state” bit lives** on classic unlock-code phones.

---

## 3) Secure boot + trust chain
Chip-level enforcement:

- Boot ROM verifies modem firmware integrity
- Prevents modified baseband images from bypassing lock

On Qualcomm devices:
- TrustZone / secure world helps protect NV items

So even if Android is modified:
> modem still enforces lock independently

---

## 4) Android framework layer (UI + restrictions)
This is what you actually interact with:

- Detects SIM state via modem APIs
- Shows:
  - “SIM not supported”
  - “Enter network unlock code”

It does NOT enforce the lock—it just reports modem status.

## 5) Carrier policy database (external layer)
For phones tied to carriers like entity["company","AT&T","telecommunications company"]:

- IMEI stored in carrier backend
- Status:

When set to “unlocked”:

- Modem NV may be updated or bypassed depending on model

## 6) OEM integration layer (Motorola side)
