## 1. `Set` / `Map` mutation risk

You’re correct — cloning before mutation handles 95% of risk. The only way this bites you is:

- `binState` (or nested structures inside it) gets passed into a helper
- That helper mutates it in place
- The reference is reused elsewhere in the same render cycle

The subtle failure mode isn’t React state corruption — it’s **mid-consolidation drift** where later decisions see partially mutated state.

If you want to bulletproof it:
- Freeze inputs in dev mode (`Object.freeze` shallowly)
- Or deep clone only at consolidation entry point and treat everything inside as mutable sandbox state

That keeps performance clean but protects you from accidental mutation creep later.

Not urgent — just future-proofing.

---

## 2. Dirty SAP exports (this is the real production landmine)

The failures won’t be obvious. They’ll be *quietly wrong math*.

### What to actively test:

**A. Quantity parsing**
- `"1,200"`  
- `"1,200.00"`  
- `"1 200"`  
- `"$1,200"`  
- `"  1200 "`  
- `"-"`  
- `""`  
- `"N/A"`

If `toNum` does `parseFloat(value)` without stripping commas and symbols, `"1,200"` becomes `1`.

That’s catastrophic and invisible.

Safer pattern:
```js
Number(String(value).replace(/[^\d.-]/g, ""))
```

---

**B. Duplicate bin + material rows**

Real exports often repeat:
- Same bin
- Same material
- Split quantities across rows

If you don’t aggregate before consolidation, you distort move math.

If you already consolidate input rows per bin/material — good.
If not, that’s the next realistic bug vector.

---

**C. Weird bin names**

Bins like:
- `R-01-01`
- `01.A.03`
- `TUNNEL_A`
- lowercase vs uppercase

If logic assumes patterns (e.g., checking prefix `R`), make sure it normalizes case first.

## 3. The O(n²) sort inside consolidation

You nailed it.

This is the only thing that could genuinely get slow.

If your loop does:

```js
while (remainingPool.length > 0) {
  remainingPool.sort(...)
  ...
}
```

That’s:

- Sort: O(n log n)
- Inside loop: up to n times
- Worst case: O(n² log n)

Hundreds of bins? Fine.
Thousands? You’ll feel it.

### Easy future fix:
Replace repeated sort with a priority structure:

- Pre-sort once
- Maintain a binary heap
- Or bucket bins by fill percentage

But do not optimize until real data justifies it.

Premature optimization is how clean code becomes spaghetti.
